1. The Limitations of Docker Containers
Main Idea
Docker solved application packaging problems but not large-scale orchestration problems.

Problems with Docker Alone
No auto-healing

No auto-scaling

Manual container management

Networking complexity

Load balancing difficult

Rolling updates hard

Multi-host management difficult

Real-Time Example
Imagine:

100 containers running

One crashes at 2 AM

Docker alone won’t restart automatically across servers

This is where Kubernetes helps.

5. Kubernetes Architecture
This is VERY IMPORTANT for interviews.

Master Node / Control Plane
Components
Component	Purpose

etcd	Database

Controller Manager	Maintains desired state

Worker Node Components
Component	Purpose

Important Interview Line
“Kubernetes follows desired state management. If the actual state differs from desired state, Kubernetes automatically corrects it.”

10. Pod in Kubernetes
VERY IMPORTANT TOPIC.

What is Pod?
Smallest deployable unit in Kubernetes.

A pod contains:

one or more containers

shared networking

shared storage

Real-Time Example

Pod
 └── Nginx Container
OR


Pod
 ├── App Container
 └── Logging Sidecar Container

12. ReplicaSets
Purpose
Maintains desired number of pod replicas.

Example
If desired replicas = 3:

One pod crashes

ReplicaSet automatically creates another

This is called:

Self-Healing

14. Horizontal Pod Autoscaler (HPA)
Purpose
Automatically scale pods based on:

CPU

Memory

Metrics

Real-Time Example
Traffic increases:

Pods increase from 2 → 10

Traffic decreases:

Pods reduce automatically

15. Readiness Probe
Purpose
Checks whether pod is ready to receive traffic.

If not ready:

Kubernetes stops sending traffic.

16. Liveness Probe
Purpose
Checks whether application is alive.

If failed:

Kubernetes restarts container.

Difference
Probe	Purpose
Readiness	Ready for traffic?
Liveness	Application alive?
